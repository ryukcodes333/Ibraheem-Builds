What I Learned Calling the Claude API for the First Time

Building Iset was my first real AI project. Prompt engineering is harder than it looks, rate limits will humble you, and the gap between a good AI product and a bad one is almost entirely in how you handle edge cases.

The First Prompt Was Terrible

My first version of Iset had a prompt that went something like: "Create a goal roadmap for: {goal}". The output was paragraphs of generic advice that you could find on any self-help blog. Useful to nobody.

The problem wasn't the model — Claude is genuinely capable of producing great structured output. The problem was that I gave it zero constraints. When you give an AI complete freedom, you get complete inconsistency.

Constraining the Output Format

The fix was telling Claude exactly what format to output. Not "here's a suggestion," but "here's the schema, output only this, nothing else." When you're building an app that needs to parse the AI's response programmatically, prose is useless. You need JSON.

javascript 
// Prompt v1 — bad
`Create a goal roadmap for: ${goal}`
// Returns: paragraphs of vague advice

// Prompt v3 — good
`Return ONLY valid JSON. No explanation. No markdown fences.

Goal: "${goal}"
Timeframe: ${weeks} weeks

Required schema:
{"milestones":[{"week":1,"title":"","tasks":[""],"resources":[""]}]}`

The difference between v1 and v3 wasn't magic — it was constraints. "Return ONLY valid JSON" tells the model what medium to use. The schema tells it what structure to follow. "No explanation. No markdown fences." eliminates the decorative text it would otherwise wrap around the output.

JSON Parsing Will Break

Even with a strict prompt, Claude occasionally outputs something that isn't valid JSON. Maybe it adds a trailing comma, or includes a comment inside the JSON, or wraps the output in backticks despite being told not to. You have to handle this.

javascript 
function safeParseRoadmap(raw) {
  // Strip markdown fences if model added them anyway
  const cleaned = raw
    .replace(/^```(?:json)?\n?/, '')
    .replace(/\n?```$/, '')
    .trim();

  try {
    return JSON.parse(cleaned);
  } catch {
    // Log and return a fallback instead of crashing
    console.error('JSON parse failed:', cleaned.slice(0, 200));
    return null;
  }
}

A try/catch around JSON.parse is not optional when your input is AI-generated. The model is good but not perfect. Your app should handle imperfect gracefully.

Rate Limits Are Real

I hit Anthropic's rate limit on the first day of testing. Turns out when you're clicking "generate" every 10 seconds while debugging, you burn through your daily limit fast.

The lesson: add request timestamps and cooldowns during development. Even a simple 3-second cooldown between requests saved me from hitting limits constantly. In production you'd handle this properly with retry-after headers, but for a personal project a basic cooldown works fine.

What Makes a Good AI Product

After building Iset, I think the quality of an AI-powered product is almost entirely determined by:

  • The prompt. Vague prompts give vague outputs. Constrained prompts give consistent, useful outputs.
  • Error handling. The model will fail sometimes. Your UI should handle that without crashing or confusing the user.
  • Loading states. AI responses take 3–10 seconds. Show a loading indicator. People will close the tab if they see nothing happening.
  • Fallbacks. What does your app do when the API is down or returns something invalid? If the answer is "it breaks," you have more work to do.

The AI part is the easy part. Prompt → response → parse → render. The hard part is everything around it.

Iset is one of the projects I'm most proud of because it actually works reliably. That's harder to achieve than it sounds.
