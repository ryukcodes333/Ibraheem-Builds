Building NexTV: An Anime Streaming Site with No Backend

No paid API, no server, no budget. Just the TMDB API, some creative caching, and a lot of edge cases I didn't see coming. This is how NexTV became something actually usable.

The Constraint

I had no money to spend on this. No hosting budget, no paid API access. Whatever I built had to run entirely in the browser using free APIs. That's a real constraint, but it's also clarifying — when you can't do everything, you focus on what you can actually do.

TMDB (The Movie Database) has a free API with solid anime metadata: posters, synopses, episode counts, scores. No actual video files, but that wasn't the goal. NexTV is a discovery and organization tool first.

Rate Limits and Debounce

TMDB allows 40 API requests per 10 seconds. That sounds like a lot until you have a search box that fires a request on every keystroke. Three letters typed quickly = three wasted requests on incomplete queries, plus you're using up your rate limit budget on useless calls.

The fix is debounce — wait until the user stops typing before sending the request.

javascript 
let searchTimer;

searchInput.addEventListener('input', (e) => {
  clearTimeout(searchTimer);

  // Only search after user pauses for 400ms
  searchTimer = setTimeout(() => {
    if (e.target.value.trim()) search(e.target.value);
  }, 400);
});

400ms is the sweet spot. Faster and it's still spammy. Slower and the search feels sluggish. I tested with 200ms, 300ms, 400ms, and 600ms before landing on 400.

Lazy Loading Posters

A browse page might show 40–60 posters at once. If you set every img src immediately, the browser tries to load all 60 images at page load. That's slow and wastes bandwidth for images the user never scrolls to.

The solution is lazy loading with IntersectionObserver. Store the real image URL in a data-src attribute, use a placeholder as src, and only swap in the real URL when the image is about to enter the viewport.

javascript 
const observer = new IntersectionObserver((entries) => {
  entries.forEach(({ isIntersecting, target }) => {
    if (isIntersecting) {
      target.src = target.dataset.src;
      observer.unobserve(target); // stop watching once loaded
    }
  });
}, { rootMargin: '200px' }); // load 200px before visible

document.querySelectorAll('img[data-src]')
  .forEach(img => observer.observe(img));

The rootMargin: '200px' tells the observer to trigger 200px before the image enters the viewport. This means images are already loading by the time the user scrolls to them — no visible pop-in.

The API Key Problem

Here's the honest truth: the TMDB API key is visible to anyone who opens DevTools. There's no way around this when everything runs in the browser with no backend.

For NexTV this is acceptable because TMDB keys are free, rate-limited, and don't have billing attached. Someone stealing the key can make the same API calls I can — there's no cost to me. For a real product handling user data or paid APIs, you'd proxy through a backend. But for a solo project with no budget, client-side is fine.

What I'd Build Differently Now

A thin backend — even a free-tier Vercel edge function — would let me hide the API key and add simple caching. Right now every user's browser makes its own API calls. With a backend, you'd cache the trending data server-side and serve it to everyone from one cached response. Fewer calls, faster for users.

I didn't build that originally because I didn't know how. I do now. That's the point of building things.

NexTV taught me more about browser performance than any article I'd read. The constraints of "no backend, no budget" forced me to actually understand how the browser handles network requests, images, and memory. Worth it.
