How I Built a WhatsApp Bot with a Full Economy System

Shadow Garden started as "I want a bot that does cool stuff." Three months later it had currency, Pokémon, and turn-based battles — all in your DMs. Here's how.

Why WhatsApp

Everyone I know uses WhatsApp. Not Discord, not Telegram — WhatsApp. If I wanted to build something people would actually interact with, it had to be there.

The problem is WhatsApp doesn't have an official bot API. So I had to use Baileys, a Node.js library that reverse-engineers the WhatsApp Web protocol. It works, but WhatsApp can ban your number if you go too hard, so you have to be careful.

Keeping the Connection Alive

The first real technical problem was session management. WhatsApp aggressively closes connections that look like automated clients. My first version would disconnect every 20–30 minutes, which made the bot useless.

The fix was a reconnect loop. When the connection drops, check if you've been logged out (if so, don't retry), otherwise call the startup function again. It recursively restarts itself. Not elegant, but it works.

javascript 
client.on('connection.update', async ({ connection, lastDisconnect }) => {
  if (connection === 'close') {
    const code = lastDisconnect?.error?.output?.statusCode;
    if (code !== DisconnectReason.loggedOut) startBot();
  }
});

Credentials are saved locally so you don't have to scan the QR code every restart. This was the part that unlocked everything — once sessions were stable, I could actually build features.

The Economy System

Every user gets a balance stored in MongoDB. The core operations are simple: check balance, add coins, subtract coins. The tricky part is making sure two commands hitting the database at the same moment don't cause race conditions.

MongoDB's findOneAndUpdate with $inc is atomic, so you can safely increment without a read-modify-write cycle. I use upsert: true everywhere so new users get created automatically on their first command instead of throwing an error.

javascript 
// Atomic coin transfer — no race condition
await User.bulkWrite([
  { updateOne: { filter: { jid: sender }, update: { $inc: { balance: -amount } } } },
  { updateOne: { filter: { jid: receiver }, update: { $inc: { balance: amount } }, upsert: true } }
]);

Pokémon and Battles

The Pokémon system uses a local JSON dataset with ~150 species, each with base stats. When a Pokémon spawns, it gets a random HP modifier so every catch is slightly different. Catches go into the user's inventory array in MongoDB.

Battles were the hardest feature. The flow is: User 1 challenges User 2 → User 2 accepts → system creates a battle document in MongoDB → turn-based loop starts. Each turn, both players pick a move, the system resolves damage, updates HP, and reports the result.

The edge case that broke things: what if someone never responds? I had to add a 60-second timeout per turn. If a player doesn't respond in time, they forfeit the round. That timeout was originally 30 seconds and people kept complaining it was too short.

What I'd Do Differently

The command parser is a mess. I'm using a giant if/else chain to match command strings, which works but is fragile. A proper command registry with a map of string → handler function would be cleaner and easier to extend. I knew this while building it and did it the bad way anyway because I just wanted to ship something.

Also: WhatsApp bots live and die by session stability. If you're building one, invest in the reconnect logic early. Everything else depends on it.
